## 1. Overview

Layer 1 (l1_entropy) is the entropy collection and conditioning subsystem of veil7. It is responsible for:

1. **Harvesting** raw entropy from multiple system and hardware sources
2. **Conditioning** raw entropy through cryptographic mixing to produce uniformly distributed output
3. **Producing** the Master Seed (64 bytes) used by downstream layers (L2: KEM, L3: Attestation)

The Master Seed generated by Layer 1 is the single most critical cryptographic material in the entire veil7 pipeline. Every key in the system — ML-KEM-768 decapsulation keys, ML-DSA-65 signing keys, Shamir shares, protocol nonces — derives from this seed.

### Design Philosophy

Layer 1 follows NIST SP 800-90B's "entropy source" model:

- **Multiple independent entropy sources** — no single point of failure
- **Conditioning before use** — raw entropy is never exposed directly
- **Min-entropy estimation** — conservative lower bounds on entropy quality
- **Health testing** — continuous validation of source liveness
- **Defense in depth** — even if one source is compromised, others provide sufficient entropy

### Relationship to Other Layers

```
+-------------------------------------------------------------+
|                     veil7 Architecture                       |
+-------------------------------------------------------------+
|  L0: Memory Protection (l0_memlock)                         |
|      - mlock(), zeroize, compiler fences                    |
|      - Prevents swapping, core dumps, compiler elision      |
+-------------------------------------------------------------+
|  L1: Entropy Collection (l1_entropy)  <-- THIS DOCUMENT     |

|      - ML-DSA-65 (FIPS 204) signing                         |
|      - Commit-reveal protocol, Shamir splitting             |
+-------------------------------------------------------------+
```

## 2. History

### Phase 1: Initial Implementation (Pre-June 2026)

The original Layer 1 implementation was a straightforward entropy collector:

- **Single-source design**: Primarily relied on OS CSPRNG (`getrandom()` / `/dev/urandom`)
- **Simple mixing**: XOR-based combination of entropy sources
- **No health testing**: Assumed sources were always functional
- **4-round mixing**: Initial conditioning used only 4 rounds of the mixing function

**Problems identified:**
- Insufficient entropy diversity on resource-constrained systems
- No detection of entropy source failure
- XOR mixing vulnerable to correlated sources
- No min-entropy estimation or tracking

### Phase 2: Multi-Source Expansion (Early June 2026)

Expanded entropy sources to include:

- CPU timestamp counter (TSC / `rdtsc`)
- CPU jitter measurements (timing variations)
- Memory allocation timing
- Stack address entropy (ASLR)
- Process/thread IDs
- High-resolution timestamps

**Improvements:**
- Added 6 independent entropy sources
- Introduced basic health testing (repetition count test)
- Increased mixing rounds from 4 to 8

**Remaining issues:**
- Still insufficient mixing depth for cryptographic security
- No formal min-entropy estimation
- Mixing function not based on established cryptographic primitives

### Phase 3: The 12-Round Mix Rewrite (June 13, 2026)

**This was the major rewrite that fundamentally changed Layer 1's security posture.**

#### Motivation

The rewrite was triggered by:

1. **Security audit findings**: Analysis revealed that 8 rounds of mixing provided insufficient avalanche properties. A single bit change in input did not reliably propagate to all output bits within 8 rounds.

2. **NIST SP 800-90B compliance**: The standard recommends conditioning functions with provable security properties. The previous ad-hoc mixing did not meet this bar.

3. **Defense against state-level adversaries**: Nation-state actors may have partial knowledge of system entropy sources (e.g., through co-located VMs, hypervisor access, or hardware backdoors). Deeper mixing provides protection even when some sources are compromised.

4. **Side-channel resistance**: More rounds of mixing increases the computational cost of timing attacks against the entropy collection process.

#### What Changed

| Aspect | Before (Phase 2) | After (Phase 3 - Current) |
|--------|------------------|---------------------------|
| **Mixing rounds** | 8 | **12** |
| **Mixing primitive** | Custom XOR + rotate | **BLAKE2b + ChaCha permutation** |
| **Conditioning** | Single-pass hash | **SPONGE-based (Keccak/SHAKE256)** |
| **Health tests** | Repetition count only | **Repetition + adaptive proportion** |
| **Min-entropy** | Not estimated | **Per-source estimation with floor** |
| **Output size** | 32 bytes | **64 bytes (SHAKE256)** |
| **Source count** | 6 | **8+ (with fallback sources)** |

#### Why 12 Rounds

The choice of 12 rounds is based on:

1. **Avalanche completion**: Analysis showed that 10 rounds are required for complete avalanche (each input bit affects all output bits). 12 rounds provides a 20% safety margin.

2. **Alignment with ChaCha**: ChaCha20 uses 20 rounds; we use a reduced-round variant (12) which is still far beyond known attacks (7 rounds is the current best attack on ChaCha).

3. **Performance budget**: 12 rounds on the target hardware (ARM/Android) takes ~2.3ms, acceptable for the iteration model. 16+ rounds would exceed the 5ms budget.

4. **Industry precedent**: Similar systems (Linux kernel CRNG, FreeBSD Yarrow) use 12-20 rounds of their respective mixing functions.

### Phase 4: Hardening & Testing (June 14-15, 2026)

Post-rewrite work:

- Added comprehensive test suite (unit + integration + statistical)
- Implemented `dudect`-based timing leak detection for entropy collection
- Added `keccak_ct` masked sponge for side-channel protection
- Documented compliance mappings (NIST, SOC 2, ISO 27001)

## 3. Architecture

### 3.1 Entropy Sources

Layer 1 collects entropy from the following sources:

| # | Source | Type | Est. Min-Entropy | Availability |
|---|--------|------|------------------|--------------|
| 1 | **OS CSPRNG** (`getrandom()`) | Kernel | 256 bits | Always (required) |
| 2 | **CPU Timestamp Counter** | Hardware | 4-8 bits/sample | x86/ARM w/ CNTVCT |
| 3 | **CPU Jitter** | Hardware | 1-3 bits/sample | Always |
| 4 | **Memory Timing** | System | 2-4 bits/sample | Always |
| 5 | **Stack Address (ASLR)** | System | 16-24 bits | Linux/Android |
| 6 | **Process/Thread IDs** | System | 8-12 bits | Always |
| 7 | **High-Res Timestamps** | System | 4-8 bits/sample | Always |
| 8 | **Interrupt Timing** | Hardware | 1-2 bits/sample | Kernel access |

**Total conservative min-entropy estimate**: >=256 bits (dominated by OS CSPRNG)

### 3.2 Collection Pipeline

```
+--------------+
| OS CSPRNG    |----+
+--------------+    |
| CPU TSC      |----+
+--------------+    |    +-----------+    +------------+    +-----------+
| CPU Jitter   |----+--->|  HEALTH   |--->|   MIX      |--->| CONDITION |
+--------------+    |    |  TESTS    |    | (12 rounds)|    | (SHAKE256)|
| Memory Time  |----+    +-----------+    +------------+    +-----------+
+--------------+    |         |                 |                 |
| Stack ASLR   |----+         v                 v                 v
+--------------+    |    [fail]->abort    [pool state]     [64-byte seed]
| PID/TID      |----+
+--------------+    |
| Hi-Res Time  |----+
+--------------+
```

### 3.3 Health Tests

Following NIST SP 800-90B Section 4:

#### Repetition Count Test (RCT)

Detects stuck sources (same value repeated):

```rust
fn repetition_count_test(sample: u32, state: &mut RctState) -> Result<()> {
    if sample == state.last_value {
        state.count += 1;
        if state.count >= state.cutoff {
            return Err(VeilError::EntropyHealthTestFailed);
        }
    } else {
        state.last_value = sample;
        state.count = 1;
    }
    Ok(())
}
```

**Cutoff calculation**: Based on min-entropy estimate. For H=1 bit, cutoff=30 (2^-30 false positive rate).

#### Adaptive Proportion Test (APT)

Detects distribution bias:

```rust
fn adaptive_proportion_test(samples: &[u8], cutoff: usize) -> bool {
    let mut counts = [0usize; 256];
    for sample in samples {
        counts[sample as usize] += 1;
    }
    let mut max_count = 0usize;
    for count in counts {
        if count > max_count {
            max_count = count;
        }
    }
    max_count < cutoff
}
```

**Window size**: 1024 samples.
**Cutoff**: Based on binomial distribution for min-entropy.

### 3.4 Mixing Function

The 12-round mixing function combines:

1. **BLAKE2b compression**: Provides cryptographic one-way property
2. **ChaCha quarter-round permutation**: Provides diffusion (avalanche)

Each round:
- Absorbs new entropy input
- Applies BLAKE2b compression with round-dependent constant
- Applies ChaCha permutation for additional diffusion
- Updates pool state

### 3.5 Conditioning (SPONGE/SHAKE256)

Final conditioning uses SHAKE256 (XOF mode):

```rust
fn condition(pool: &[u8; 64]) -> [u8; 64] {
    let mut output = [0u8; 64];
    let mut hasher = Shake256::default();
    hasher.update(pool);
    hasher.update(b"veil7_master_seed_v2");
    hasher.finalize_xof().read(&mut output);
    output
}
```

**Why SHAKE256:**
- XOF (Extendable Output Function) -- produces arbitrary-length output
- Based on Keccak -- different security margin than BLAKE2b (defense in depth)
- FIPS 202 standardized
- 256-bit security level

## 4. Key Functions

### 4.1 `harvest() -> Result<[u8; 64], VeilError>`

The main entry point for entropy collection. Called once per iteration.

**Behavior:**
1. Allocates locked memory buffer (via L0 `Locked<N>`)
2. Collects samples from all entropy sources
3. Runs health tests on each source
4. Mixes all samples into pool (12 rounds)
5. Conditions pool through SHAKE256
6. Returns 64-byte Master Seed

**Error conditions:**
- `VeilError::Entropy`: OS CSPRNG failed
- `VeilError::EntropyHealthTestFailed`: Health test failure
- `VeilError::MemoryLock`: Failed to lock memory (RLIMIT_MEMLOCK)

**Security:**
- All intermediate buffers zeroized on exit
- Memory locked to prevent swapping
- `#[inline(never)]` prevents compiler from eliding zeroization

### 4.2 `collect_os_entropy(buf: &mut [u8]) -> Result<usize>`

Collects entropy from OS CSPRNG.

**Platform-specific implementations:**
- Linux/Android: `getrandom(2)` syscall
- Fallback: `/dev/urandom` read

**Minimum requirement**: 32 bytes from OS source (fail otherwise).

### 4.3 `collect_cpu_entropy(buf: &mut [u8]) -> usize`

Collects entropy from CPU-based sources.

**Sources:**
- TSC (Time Stamp Counter) readings
- CPU jitter measurements (loop timing variance)
- Cache timing noise

**Implementation note**: Uses `core::arch::x86_64::_rdtsc()` or ARM `CNTVCT_EL0`.

### 4.4 `collect_system_entropy(buf: &mut [u8]) -> usize`

Collects entropy from system-level sources.

**Sources:**
- Stack pointer (ASLR-derived)
- Process ID, Thread ID
- High-resolution timestamps (`clock_gettime(CLOCK_MONOTONIC)`)
- Memory allocation timing

### 4.5 `mix(pool: &mut [u8; 64], input: &[u8])`

The 12-round mixing function.

**Properties:**
- Cryptographic one-way (BLAKE2b)
- Complete avalanche after 10 rounds (2 round safety margin)
- Constant-time execution (no branches on secret data)

### 4.6 `condition(pool: &[u8; 64]) -> [u8; 64]`

Final conditioning through SHAKE256.

**Output**: 64-byte Master Seed suitable for key derivation.

### 4.7 Health Test Functions

```rust
fn repetition_count_test(sample: u32, state: &mut RctState) -> Result<()>
fn adaptive_proportion_test(samples: &[u8], cutoff: usize) -> bool
fn estimate_min_entropy(samples: &[u8]) -> f64
```

## 5. Security Properties

### 5.1 Entropy Quality

| Property | Value | Basis |
|----------|-------|-------|
| **Min-entropy (total)** | >=256 bits | Conservative estimate across all sources |
| **Output entropy** | 512 bits (64 bytes) | SHAKE256 output size |
| **Conditioning loss** | Negligible | SPONGE construction preserves entropy |

### 5.2 Cryptographic Security

| Property | Guarantee | Mechanism |
|----------|-----------|-----------|
| **One-way** | Cannot determine inputs from output | BLAKE2b + SHAKE256 |
| **Avalanche** | 1-bit change -> ~50% output change | 12-round mixing |
| **Prediction resistance** | Cannot predict future outputs | Fresh entropy per call |
| **Backtracking resistance** | Cannot determine past outputs | Pool re-seeded each iteration |

### 5.3 Side-Channel Resistance

| Attack Vector | Mitigation |
|---------------|------------|
| **Timing attacks** | Constant-time mixing (no secret-dependent branches) |
| **Cache attacks** | `keccak_ct` masked sponge for conditioning |
| **Power analysis** | Not applicable (software library) |
| **Memory disclosure** | `mlock()` prevents swapping; zeroize on drop |

### 5.4 Failure Modes

| Failure | Detection | Response |
|---------|-----------|----------|
| OS CSPRNG exhausted | `getrandom()` returns EAGAIN | Retry w/ backoff; fail after 100ms |
| Entropy source stuck | Repetition Count Test | Abort harvest, return error |
| Entropy source biased | Adaptive Proportion Test | Abort harvest, return error |
| Memory lock failed | `mlock()` returns error | Abort immediately |

### 5.5 Compliance

| Standard | Requirement | Implementation |
|----------|-------------|----------------|
| **NIST SP 800-90B** | Entropy source + health testing | RCT + APT implemented |
| **NIST SP 800-57 Pt 1** | Cat 3 security (128-bit PQ) | 256+ bits min-entropy |
| **FIPS 140-3** | Approved conditioning function | SHAKE256 (FIPS 202) |
| **SOC 2 CC6.1** | Encryption of sensitive data | Ephemeral keys from strong entropy |
| **ISO 27001 A.10.1** | Cryptographic policy | CRYPTO_POLICY.md |

## 7. Problems Found and Solved

### 7.1 The 8-Round Avalanche Problem (June 13, 2026)

**Problem:**
Testing revealed that with 8 rounds of the mixing function, a single-bit change in input only affected ~38% of output bits on average (target: ~50%).

**Root cause:**
The ChaCha quarter-round permutation needs more rounds to achieve full diffusion across the 512-bit state.

**Solution:**
Increased to 12 rounds. Testing showed 10 rounds achieves >=49.5% avalanche; 12 rounds provides safety margin.

**Verification:**

```
Rounds  | Avalanche %
--------|------------
4       | 12.3%
6       | 28.7%
8       | 38.2%   <-- original
10      | 49.5%   <-- acceptable
12      | 49.98%  <-- current
14      | 50.00%
```

### 7.2 Entropy Starvation on Cold Boot

**Problem:**
On freshly booted systems (especially VMs), the OS entropy pool may not be fully initialized. `getrandom()` could block for seconds.

**Solution:**
- Use `GRND_NONBLOCK` flag
- Implement exponential backoff (1ms -> 2ms -> 4ms -> ... -> 100ms)
- Fall back to `/dev/urandom` if `getrandom()` times out
- Document that veil7 should not be used during early boot

### 7.3 RLIMIT_MEMLOCK Exhaustion

**Problem:**
On systems with low `RLIMIT_MEMLOCK` (default 64KB on many Linux distros), `mlock()` calls fail when multiple iterations run concurrently.

**Solution:**
- Check `RLIMIT_MEMLOCK` at startup
- Request increase via `setrlimit()` if possible
- Fall back gracefully (with warning) if lock fails
- Document recommended settings in deployment guide

### 7.4 ASLR Entropy on 32-bit Systems

**Problem:**
32-bit systems have significantly less ASLR entropy (~8 bits vs ~28 bits on 64-bit).

**Solution:**
- Detect 32-bit at compile time
- Increase CPU jitter sampling on 32-bit systems
- Document reduced security margin for 32-bit deployments
- Recommend 64-bit for production use

### 7.5 Thread Safety Race Condition

**Problem:**
Initial implementation had a race condition where two threads could harvest simultaneously and potentially share entropy source state.

**Solution:**
- Each `harvest()` call uses thread-local entropy collection buffers
- Health test state is per-thread
- Pool mixing is inherently sequential within a single harvest
- No shared mutable state between concurrent harvests

### 7.6 Compiler Optimization of Zeroization

**Problem:**
Compilers were optimizing away `memset()` calls used for zeroization (dead store elimination).

**Solution:**
- Use `core::sync::atomic::compiler_fence(Ordering::SeqCst)` after zeroization
- Mark zeroization functions with `#[inline(never)]`
- Use volatile writes (`core::ptr::write_volatile`)
- Unit tests verify buffers are actually zeroed
