## 1. Overview

Layer 5 (`l5_verify`) is the **verification layer** of the veil7 7-layer
stateless post-quantum verification engine. It receives the cryptographic
proof artifacts generated by Layer 4 (`l4_prove`) and performs
**constant-time dual verification** to determine whether the attestation
is valid.

Layer 5 executes two independent verification checks:

1. **`sig_ok`** — ML-DSA-65 (FIPS 204) digital signature verification
2. **`kem_ok`** — ML-KEM-768 (FIPS 203) KEM round-trip verification

Both checks must pass for the verdict to be `valid`. The checks are
performed in **constant time** using `subtle::Choice` rather than native
booleans, preventing timing side-channel leakage of intermediate
verification results.

### 1.1 Role in the Pipeline

```
L0 (mlock)     → Lock seed material in RAM, prevent swap
L1 (entropy)   → Harvest fresh OS CSPRNG entropy → 64-byte master seed
L2 (keygen)    → Derive ephemeral ML-KEM-768 + ML-DSA-65 keypairs from seed
L3 (commit)    → Domain-separated SHAKE256 commitment binding identity + claim
L4 (prove)     → Generate post-quantum proof (ML-DSA-65 signature + KEM round-trip)
L5 (verify)    → ★ Constant-time verification (dual check: sig_ok & kem_ok) ★
L6 (zeroise)   → Explicit scrub barrier — wipe all secrets
L7 (emit)      → Emit traceless Verdict (validity bit + transcript hash)
```

Layer 5 is the **security gate** of the entire pipeline. It is the point
at which the system decides whether the proof produced by Layer 4 is
authentic and bound to the correct ephemeral identity and claim. A
failure at Layer 5 means the attestation is rejected — no Verdict is
emitted, and all intermediate material is zeroized by Layer 6.

### 1.2 Design Philosophy

| Philosophy | Application in L5 |
|------------|-------------------|
| **Stateless** | Verification uses only the ephemeral keys from the current iteration |
| **Math over abstraction** | Direct libcrux API calls; no unnecessary wrapper layers |
| **Defence-in-depth** | Dual independent checks (signature + KEM); both must pass |
| **Constant-time** | `subtle::Choice` for all comparisons; no secret-dependent branches |
| **No trace** | Verification state exists only on the stack; zeroized on function exit |
| **Refuse > guess** | Malformed inputs are rejected before cryptographic operations |

### 1.3 Inputs and Outputs

**Inputs (from Layer 4):**

| Input | Type | Size | Source |
|-------|------|------|--------|
| ML-DSA-65 verification key | `&[u8]` | 1952 bytes | L2 keygen |
| ML-DSA-65 signature | `&[u8]` | 3309 bytes | L4 prove |
| Signed message (commitment) | `&[u8]` | 32 bytes | L3 commit |
| ML-KEM-768 encapsulation key | `&[u8]` | 1184 bytes | L2 keygen |
| ML-KEM-768 decapsulation key | `&[u8]` | 2400 bytes | L2 keygen |
| ML-KEM-768 ciphertext | `&[u8]` | 1088 bytes | L4 prove |
| ML-KEM-768 shared secret (encapsulated) | `&[u8]` | 32 bytes | L4 prove |

**Output:**

| Output | Type | Description |
|--------|------|-------------|
| `sig_ok` | `subtle::Choice` | ML-DSA-65 signature verification result |
| `kem_ok` | `subtle::Choice` | ML-KEM-768 round-trip verification result |
| `valid` | `subtle::Choice` | `sig_ok & kem_ok` — combined verdict |

## 2. Complete History

### 2.1 Phase 1: Initial Implementation (June 10, 2026)

The initial Layer 5 implementation provided basic verification using
RustCrypto's post-quantum crates:

- ML-DSA-65 signature verification via `pqcrypto-dilithium`
- ML-KEM-768 KEM round-trip via `pqcrypto-kyber`
- Boolean return values for verification results
- Basic input validation (size checks)

**Characteristics:**
- Functional but not formally verified
- Boolean comparison vulnerable to timing leakage
- No constant-time guarantees
- RustCrypto backends lacked formal CT proofs

**Remaining issues:**
- Boolean `verify` leaked timing information about intermediate state
- RustCrypto implementations were not formally verified for constant-time
- No hardening tests to enforce CT discipline
- Potential KyberSlash-class vulnerabilities in compression/decompression

### 2.2 Phase 1.1: Migration to libcrux (June 13–14, 2026)

A critical migration replaced the RustCrypto post-quantum backends with
**libcrux** (Cryspen), which is formally verified via hax/F* and
constant-time by construction.

| Aspect | RustCrypto | libcrux |
|--------|-----------|---------|
| Verification | None | hax/F* formally verified |
| Constant-time | Assumed | Proven by construction |
| KyberSlash risk | Possible | Eliminated |
| T-table exposure | Yes (sha3) | No (libcrux-sha3) |
| ACVP validation | Not tested | Byte-perfect NIST match |

**Changes made:**
- Replaced `pqcrypto-kyber` with `libcrux-ml-kem 0.0.9` for KEM round-trip
- Replaced `pqcrypto-dilithium` with `libcrux-ml-dsa 0.0.9` for signature verify
- Updated all verification call sites
- Validated against official NIST ACVP test vectors

### 2.3 Phase 1.2: NIST ACVP Validation (June 14, 2026)

Official NIST ACVP test vectors (sourced from BoringSSL/Google →
`usnistgov/ACVP-Server`) were used to validate the verification

- Used `ConstantTimeEq` for shared secret comparison in KEM round-trip

## 3. What Was Changed and Why

### 3.1 Replacement of RustCrypto with libcrux

**What:** Replaced `pqcrypto-kyber` and `pqcrypto-dilithium` with
`libcrux-ml-kem 0.0.9` and `libcrux-ml-dsa 0.0.9`.

**Why:** The RustCrypto post-quantum crates lack formal verification and
may contain timing side-channels. The KyberSlash vulnerability (2023)
demonstrated that secret-dependent division operations in ML-KEM
compression/decompression could leak key material through timing.
libcrux is:
- Formally verified via hax/F* (mathematical proof of correctness)
- Constant-time by construction (no secret-dependent branches)
- Validated against official NIST ACVP test vectors (byte-perfect match)

### 3.2 Boolean → `subtle::Choice` Migration

**What:** Changed verification return type from `bool` to `subtle::Choice`.

**Why:** Native boolean verification creates a branch point:
```rust
// VULNERABLE: timing leak
if verify_signature(vk, sig, msg) {
    // branch taken → timing observable
}
```

The `subtle::Choice` type uses bitwise operations that execute in
constant time regardless of the verification outcome:
```rust
// CONSTANT-TIME: no branch
let sig_ok = verify_signature_ct(vk, sig, msg);  // Choice
let kem_ok = verify_kem_ct(dk, ct, ss);           // Choice
let valid = sig_ok & kem_ok;                       // bitwise AND
```

### 3.3 Addition of Input Validation

**What:** Added format and size validation for all inputs before
cryptographic operations.

**Why:** Malformed inputs could cause:
- Panics in libcrux (denial of service)
- Undefined behavior in edge cases
- Wasted computation on clearly invalid data

**Validations added:**
- ML-DSA-65 verification key: exactly 1952 bytes
- ML-DSA-65 signature: exactly 3309 bytes
- ML-KEM-768 encapsulation key: exactly 1184 bytes
- ML-KEM-768 decapsulation key: exactly 2400 bytes
- ML-KEM-768 ciphertext: exactly 1088 bytes
- ML-KEM-768 shared secret: exactly 32 bytes
- Commitment: exactly 32 bytes

### 3.4 Addition of `test_no_bool_verify` Hardening Test

**What:** A compile-time source scan test that verifies no boolean
`verify` functions exist in the codebase.

**Why:** Prevents regression to vulnerable boolean verification. The
test scans `src/layers/l5_verify.rs` for patterns like `-> bool` in
verification function signatures and fails the build if found.

### 3.5 SHAKE256 Migration for Transcript Hashing

**What:** Replaced `sha3 0.10` with `libcrux-sha3 0.0.9` for all
SHAKE256 operations used in Fiat-Shamir transcript computation.

**Why:** The `sha3` crate uses T-table (lookup-table) based Keccak.
Per-call table access patterns can leak absorbed secrets on shared-cache
hardware via Flush+Reload or Prime+Probe attacks. The 2023 Raccoon
attack exploited this class of vulnerability against ML-KEM, and a 2025
Cloudflare reduced-round Keccak regression confirmed the ongoing risk.

### 3.6 Dual-Check Architecture

**What:** Layer 5 requires BOTH `sig_ok` AND `kem_ok` to pass.

**Why:** Defense-in-depth. Each check provides independent security:

| Check | What It Proves | Attack Prevented |
|-------|---------------|-----------------|
| `sig_ok` | The commitment was signed by the ephemeral ML-DSA-65 key | Signature forgery, claim substitution |
| `kem_ok` | The prover possessed the ML-KEM-768 decapsulation key | Key substitution, replay attacks |

An adversary would need to break BOTH ML-DSA-65 EUF-CMA security AND
ML-KEM-768 IND-CCA2 security simultaneously to forge a valid attestation
that passes Layer 5.

## 4. Key Functions and Their Purposes

### 4.1 `verify_signature(vk, signature, message) -> Choice`

**Purpose:** Verifies an ML-DSA-65 (FIPS 204) digital signature in
constant time.

**Parameters:**
- `vk: &[u8]` — ML-DSA-65 verification key (1952 bytes)
- `signature: &[u8]` — ML-DSA-65 signature (3309 bytes)
- `message: &[u8]` — The signed message (L3 commitment, 32 bytes)

**Returns:** `subtle::Choice` — `1` if valid, `0` if invalid

**Implementation:** Delegates to `libcrux-ml-dsa 0.0.9` which is
formally verified (hax/F*) to be constant-time and functionally correct
per FIPS 204.

**Security:** The signature covers the L3 commitment, which binds the
claim, KEM encapsulation key, and signature verification key. Any
modification to the claim or keys invalidates the signature.

### 4.2 `verify_kem_roundtrip(dk, ciphertext, expected_ss) -> Choice`

**Purpose:** Verifies ML-KEM-768 (FIPS 203) KEM round-trip correctness
in constant time.

**Parameters:**
- `dk: &[u8]` — ML-KEM-768 decapsulation key (2400 bytes)
- `ciphertext: &[u8]` — ML-KEM-768 ciphertext (1088 bytes)
- `expected_ss: &[u8]` — Expected shared secret from encapsulation (32 bytes)

**Returns:** `subtle::Choice` — `1` if decapsulated secret matches, `0` otherwise

**Process:**
1. Decapsulate the ciphertext using the decapsulation key → `actual_ss`
2. Compare `actual_ss` with `expected_ss` using `ConstantTimeEq`
3. Return result as `subtle::Choice`

**Implementation:** Decapsulation uses `libcrux-ml-kem 0.0.9` (formally
verified). Comparison uses `subtle::ConstantTimeEq` to prevent timing
leakage of partial matches.

**Security:** This proves the prover possessed the correct decapsulation
key corresponding to the encapsulation key committed in L3. Without the
decapsulation key, an adversary cannot produce a ciphertext that
decapsulates to the correct shared secret.

### 4.3 `verify(commitment, vk, signature, ek, dk, ciphertext, shared_secret) -> Choice`

**Purpose:** The main verification entry point. Performs dual-check
verification and returns the combined verdict.

**Process:**
1. Validate all input sizes and formats
2. Compute `sig_ok = verify_signature(vk, signature, commitment)`
3. Compute `kem_ok = verify_kem_roundtrip(dk, ciphertext, shared_secret)`
4. Return `valid = sig_ok & kem_ok` (bitwise AND, constant-time)

**Returns:** `subtle::Choice` — the combined verification verdict

**Security properties:**
- Both checks execute regardless of the outcome of either (no early return)
- The bitwise AND ensures the combined result leaks nothing about
  individual check outcomes
- All intermediate `Choice` values are stack-allocated and zeroized on
  function exit

### 4.4 `validate_inputs(...)` (Internal)

**Purpose:** Pre-flight validation of all cryptographic inputs before
any expensive operations.

**Checks:**
- All byte slices have correct lengths per FIPS 203/204
- No all-zero keys (indicates generation failure)
- No obviously malformed data

**Error handling:** Returns `Err(VeilError::InvalidInput)` on failure,
preventing wasted cryptographic computation and potential panics.

### 4.5 `verify_batch(items: &[VerifyItem]) -> Vec<Choice>`

**Purpose:** Batch verification of multiple attestations for throughput.

**Process:** Iterates over items, calling `verify()` on each. Each
verification is independent and constant-time.

**Use case:** High-throughput scenarios where multiple attestations are
verified in sequence (e.g., batch auditing, log verification).

**Note:** Each verification in the batch is fully independent — no
shared state, no batch-optimized algorithms that could introduce
cross-item timing channels.

## 5. Security Properties

### 5.1 Verification Soundness

**Definition:** If Layer 5 returns `valid = 1`, then with overwhelming
probability:
1. The commitment was generated from the correct claim, KEM ek, and SIG vk
2. The signature was produced by the corresponding ML-DSA-65 signing key
3. The KEM ciphertext was produced by encapsulating to the correct ek
4. All keys were derived from the same ephemeral master seed

**Soundness argument:**
- **Signature soundness** relies on ML-DSA-65 EUF-CMA security (FIPS 204,
  NIST Category 3, ~192-bit post-quantum security). An adversary cannot
  forge a valid signature without the signing key.
- **KEM soundness** relies on ML-KEM-768 IND-CCA2 security (FIPS 203,
  NIST Category 3, ~192-bit post-quantum security). An adversary cannot
  produce a ciphertext that decapsulates to the correct shared secret
  without the encapsulation key.
- **Combined soundness** requires breaking both primitives simultaneously,
  providing defense-in-depth. Even if one primitive were compromised
  (e.g., a future attack on ML-DSA), the other (ML-KEM) would still
  protect the verification.

**Soundness error:** Negligible — bounded by the sum of the forgery
probability of ML-DSA-65 and the encapsulation failure probability of
ML-KEM-768, both of which are < 2^{-128}.

### 5.2 Constant-Time Execution

**Definition:** The execution time of Layer 5 verification is independent
of all secret values (keys, signatures, shared secrets, verification
outcomes).

**Mechanisms:**

| Component | CT Source | Verification Method |
|-----------|-----------|-------------------|
| ML-DSA-65 verify | libcrux-ml-dsa (hax/F* verified) | Formal verification by construction |
| ML-KEM-768 decaps | libcrux-ml-kem (hax/F* verified) | Formal verification by construction |
| Shared secret comparison | `subtle::ConstantTimeEq` | Bitwise XOR accumulation |
| Verdict combination | `subtle::Choice` bitwise AND | No branch on secret |
| SHAKE256 (transcript) | libcrux-sha3 (hax/F* verified) | No T-tables; formal CT |

**What this means:**
- No secret-dependent branches in any verification path
- No secret-dependent memory access patterns
- No early returns based on intermediate verification results
- Both `sig_ok` and `kem_ok` are always computed, even if one fails
- The final verdict is combined via bitwise AND, not logical AND

**Remaining concern:** The `slh-dsa` (RustCrypto) alternative backend
still uses `sha3` internally. This affects only the SLH-DSA alternative
backend, not the primary ML-KEM/ML-DSA verification path through Layer 5.

### 5.3 Dual Checks (Defense-in-Depth)

**Definition:** Layer 5 performs two cryptographically independent
verification checks, both of which must pass.

**Rationale:**

| Check | Primitive | Security Property | Failure Mode Detected |
|-------|-----------|-------------------|----------------------|
| `sig_ok` | ML-DSA-65 | EUF-CMA (existential unforgeability) | Forged signature, wrong key, modified claim |
| `kem_ok` | ML-KEM-768 | IND-CCA2 (indistinguishability) | Wrong decapsulation key, corrupted ciphertext |

**Independence:** The two checks use different mathematical structures:
- ML-DSA-65 is based on the Module-Lattice Digital Signature problem
  (hardness of finding short vectors in module lattices with rejection
  sampling)
- ML-KEM-768 is based on the Module-Lattice Key Encapsulation problem
  (hardness of Module-LWE and Module-LWR)

An attack that breaks one does not automatically break the other.

**Binding to L3 commitment:** Both checks verify data that is bound by
the Layer 3 commitment:
- The ML-DSA-65 signature covers the commitment (which includes the SIG vk)
- The KEM round-trip uses keys committed in L3 (the KEM ek is in the commitment)

This means an adversary cannot substitute keys or claims after the
commitment is made — both Layer 5 checks would fail.

### 5.4 Completeness

**Definition:** If the proof was honestly generated (correct claim,
correct keys, correct signature, correct KEM round-trip), then Layer 5
returns `valid = 1` with probability 1.

**Mechanism:** libcrux's ML-DSA-65 and ML-KEM-768 implementations are
formally verified to be functionally correct — an honestly generated
signature always verifies, and an honestly performed KEM round-trip
always produces matching shared secrets.

### 5.5 Post-Quantum Security

**Definition:** Layer 5's verification security does not rely on any
assumption vulnerable to quantum algorithms (Shor's algorithm).

**Mechanism:**
- ML-DSA-65 security rests on the hardness of the Module-Lattice
  Signature problem — no known quantum algorithm provides significant
  speedup beyond generic Grover search
- ML-KEM-768 security rests on the hardness of Module-LWE/LWR —
  similarly resistant to known quantum attacks
- The `subtle` crate's constant-time operations use only bitwise
  arithmetic — no number-theoretic assumptions

**Security level:** NIST Category 3 (~192-bit classical security,
~128-bit post-quantum security for both ML-DSA-65 and ML-KEM-768).

### 5.6 Non-Malleability of Verdict

**Definition:** An adversary cannot modify a valid proof to produce a
different valid proof that passes Layer 5 verification.

**Mechanism:**
- The ML-DSA-65 signature is non-malleable (EUF-CMA security)
- The KEM ciphertext is non-malleable (IND-CCA2 security)
- The combined verdict requires both to pass — modifying either
  component invalidates the corresponding check
- The Fiat-Shamir transcript (verified in L5) includes the protocol
  label, preventing cross-relation replay

### 5.7 Zero-Knowledge of Verification

**Definition:** The verification process itself leaks no information
about the secret key material beyond the single-bit validity verdict.

**Mechanism:**
- Constant-time execution prevents timing leakage
- The `subtle::Choice` return type prevents branch-based leakage
- Shared secret comparison uses `ConstantTimeEq` — no partial match info
- All intermediate values are stack-allocated and zeroized by Layer 6
- No logging, no error messages that distinguish between sig failure
  and KEM failure

## 7. Problems Found and Solved

### 7.1 Boolean Verification Timing Leak (HIGH — Resolved)

**Problem:** The initial implementation used `bool` return values for
verification. This creates a branch point that leaks timing information:
a valid signature takes a different code path than an invalid one,
observable via cache-timing or branch-prediction attacks.

**Solution:** Replaced all verification return types with `subtle::Choice`.
Added `test_no_bool_verify` hardening test that scans source code for
`-> bool` in verification functions and fails the build if found.

**Status:** ✅ Resolved. All verification functions return `subtle::Choice`.

### 7.2 RustCrypto PQ Implementation Risks (HIGH — Resolved)

**Problem:** Initial implementation used RustCrypto's post-quantum
crates (`pqcrypto-kyber`, `pqcrypto-dilithium`) which lack formal
verification and may contain timing side-channels (KyberSlash-class
vulnerabilities in compression/decompression).

**Solution:** Replaced with libcrux (Cryspen) which is:
- Formally verified via hax/F*
- Constant-time by construction
- Validated against NIST ACVP test vectors

**Status:** ✅ Resolved. libcrux ML-KEM and ML-DSA are formally CT.

### 7.3 T-Table Cache-Timing in Transcript Hashing (HIGH — Resolved)

**Problem:** The `sha3 0.10` crate uses T-table (lookup-table) Keccak
for SHAKE256. When computing the Fiat-Shamir transcript challenge that
Layer 5 verifies against, table access patterns could leak absorbed
secrets on shared-cache hardware (Flush+Reload / Prime+Probe).

**Solution:** Migrated to `libcrux-sha3 0.0.9` which uses a generic
Keccak implementation with **no T-tables**. Retained `keccak_ct.rs`
defense-in-depth masking layer.

**Status:** ✅ Resolved at base level. `keccak_ct.rs` retained as
defense-in-depth.

### 7.4 Early Return on Signature Failure (MEDIUM — Resolved)

**Problem:** An early optimization returned immediately if signature
verification failed, skipping KEM round-trip verification. This leaked
whether the signature check passed or failed through timing.

**Solution:** Both checks now always execute regardless of intermediate
results. The combined verdict is computed as `sig_ok & kem_ok` (bitwise
AND of `subtle::Choice` values) after both checks complete.

**Status:** ✅ Resolved. No early returns in verification path.

### 7.5 Input Validation Missing (MEDIUM — Resolved)

**Problem:** No validation of input sizes before passing to libcrux.
Malformed inputs could cause panics or undefined behavior.

**Solution:** Added `validate_inputs()` pre-flight check that verifies
all byte slices have correct lengths per FIPS 203/204, rejects all-zero
inputs, and returns `Err(VeilError::InvalidInput)` on failure.

**Status:** ✅ Resolved. All inputs validated before cryptographic ops.

### 7.6 Partial Match Leakage in KEM Comparison (LOW — Resolved)

**Problem:** Initial KEM round-trip comparison used byte-by-byte `==`
which could short-circuit on first mismatch, leaking which byte position
differed.

**Solution:** Replaced with `subtle::ConstantTimeEq` which XORs all
bytes and accumulates the result, ensuring comparison time is
independent of where (or whether) the values differ.

**Status:** ✅ Resolved. All comparisons use `ConstantTimeEq`.

### 7.7 `#[inline(never)]` on Trait Impl (LOW — Resolved)

**Problem:** An early enhancement pass placed `#[inline(never)]` on a
trait impl block rather than on individual functions, causing a
compilation error.

**Solution:** Moved `#[inline(never)]` to individual function
definitions within the Drop impl.

**Status:** ✅ Resolved.
